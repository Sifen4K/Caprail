## Context

从零构建一款 Windows 截图录屏工具，对标 PixPin。目标用户为团队内部同事，需提供安装包分发。项目无历史代码包袱，可自由选择技术栈。

核心约束：
- Windows 10+ only
- 性能优先（Rust）
- 需要安装包分发和自动更新
- 个人主导开发，需控制复杂度

## Goals / Non-Goals

**Goals:**
- 提供流畅的截图体验（区域/窗口/全屏），响应延迟 < 100ms
- 提供丰富的标注工具，覆盖日常工作所需
- 提供高性能屏幕录制，支持 60fps，CPU 占用低
- 录屏导出 MP4/GIF，支持简单剪辑
- 截图可钉在屏幕上，支持 OCR 文字提取
- 一键安装，自动更新

**Non-Goals:**
- 不做跨平台（macOS / Linux）
- 不做复杂视频编辑（多轨、转场、特效）
- 不做云端同步/云存储
- 不做滚动截图（技术复杂度高，优先级低）
- 不做录音/摄像头叠加（第一版）

## Decisions

### 1. 应用框架：Tauri v2

**选择**: Tauri v2 (Rust 后端 + WebView2 前端)

**替代方案**:
- Electron: 内存占用大（~150MB+），打包体积大（~200MB+），性能差
- 纯 Rust GUI (iced/egui): 图形标注编辑器开发效率极低，缺少成熟的 Canvas 2D 编辑生态
- WPF/C#: 性能好但不是 Rust，团队技术偏好不符

**理由**: Tauri v2 打包小（~5MB），性能关键路径在 Rust 侧不受 WebView 影响，标注编辑器用 HTML Canvas 实现效率最高，Windows 上使用系统自带 WebView2。

### 2. 前端框架：Vanilla TypeScript + Canvas API

**选择**: 不使用 React/Vue 等框架，用原生 TypeScript + HTML Canvas

**替代方案**:
- React/Vue: 引入虚拟 DOM 开销，对 Canvas 密集操作无优势
- Fabric.js/Konva.js: Canvas 库，可考虑用于标注编辑器

**理由**: 应用 UI 较简单（托盘菜单 + 几个功能窗口），不需要复杂的组件化框架。标注编辑器可能引入 Fabric.js 简化开发。录屏剪辑用原生 `<video>` + 自定义时间轴控件。

### 3. 截屏引擎：DXGI Desktop Duplication API

**选择**: Windows DXGI Desktop Duplication API (通过 windows-rs crate)

**替代方案**:
- GDI BitBlt: 老旧，不支持硬件加速内容，性能差
- Windows.Graphics.Capture (WGC): UWP API，需要用户授权弹窗，体验差
- PrintWindow: 仅支持单窗口，无法区域截图

**理由**: DXGI 是 Windows 10+ 上性能最好的屏幕捕获方案，直接访问 GPU 显存，延迟最低。

### 4. 录屏编码：ffmpeg (通过进程调用)

**选择**: 以子进程方式调用 ffmpeg，将原始帧通过 stdin pipe 传入

**替代方案**:
- ffmpeg C 绑定 (ffmpeg-sys): 编译复杂，API 不稳定
- Media Foundation: Windows 原生但 API 复杂，文档差
- 纯 Rust 编码器 (rav1e/x264 bindings): 生态不成熟

**理由**: ffmpeg 子进程方案最简单可靠。通过 pipe 传入原始帧，ffmpeg 负责编码输出。避免了复杂的 C 绑定编译问题。GIF 输出也通过 ffmpeg 的 palettegen + paletteuse 滤镜实现。ffmpeg 二进制随安装包分发。

### 5. OCR 引擎：PaddleOCR

**选择**: PaddleOCR (通过 PaddleOCR CLI 或 ONNX Runtime 推理)

**替代方案**:
- Windows OCR (WinRT): 中文识别质量一般
- Tesseract: 中文效果差
- 在线 API (百度/腾讯): 需要网络，有隐私顾虑

**理由**: PaddleOCR 中英文识别质量最好，可离线运行。使用 ONNX Runtime 加载模型做推理，避免依赖 PaddlePaddle 框架。模型文件随安装包分发。

### 6. 窗口架构

```
┌──────────────────────────────────────────────────┐
│                 主进程 (Rust)                      │
│  ┌────────────┐  ┌────────────┐  ┌────────────┐ │
│  │ 系统托盘    │  │ 全局热键    │  │ 截屏引擎   │  │
│  │ 管理器     │  │ 管理器     │  │ (DXGI)     │  │
│  └────────────┘  └────────────┘  └────────────┘ │
│  ┌────────────┐  ┌────────────┐  ┌────────────┐ │
│  │ 录屏引擎    │  │ OCR 引擎   │  │ ffmpeg     │  │
│  │ (DXGI)     │  │ (ONNX RT)  │  │ 进程管理    │  │
│  └────────────┘  └────────────┘  └────────────┘ │
├──────────────────────────────────────────────────┤
│                 窗口 (WebView)                     │
│  ┌────────────┐  ┌────────────┐  ┌────────────┐ │
│  │ 截图选区    │  │ 标注编辑器  │  │ 录屏剪辑   │  │
│  │ 覆盖层窗口  │  │ 窗口       │  │ 窗口       │  │
│  │ (全屏透明)  │  │            │  │            │  │
│  └────────────┘  └────────────┘  └────────────┘ │
│  ┌────────────┐  ┌────────────┐                  │
│  │ 钉图浮窗    │  │ 设置窗口   │                  │
│  │ (多实例)    │  │            │                  │
│  └────────────┘  └────────────┘                  │
└──────────────────────────────────────────────────┘
```

**截图选区覆盖层**: 全屏无边框透明窗口，覆盖所有显示器。用户在上面拖拽选区。选区确认后截取对应区域像素。

**标注编辑器**: 独立窗口，接收截图数据，Canvas 上叠加标注图层。

**钉图浮窗**: 多实例窗口，每次钉图创建一个新的 always-on-top 窗口。

### 7. 数据流

```
截图流程:
  快捷键 → DXGI 全屏抓取 → 选区覆盖层(用户选区)
    → 裁剪像素 → 标注编辑器(用户标注)
    → 输出: 复制剪贴板 / 保存文件 / 钉在屏幕 / OCR

录屏流程:
  快捷键 → 选区覆盖层(用户选区) → 开始录制
    → DXGI 帧循环 → pipe → ffmpeg 编码
    → 停止录制 → 临时文件
    → 剪辑窗口(用户裁剪/调速) → 导出 MP4/GIF
```

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| DXGI 在某些驱动/GPU 上不稳定 | 截图/录屏失败 | 降级到 GDI BitBlt 作为 fallback |
| ffmpeg 子进程方式有启动开销 | 录屏开始有短暂延迟 | 预启动 ffmpeg 进程，等待 pipe 输入 |
| PaddleOCR ONNX 模型体积大（~10MB） | 安装包体积增大 | 可接受，总包控制在 50MB 内 |
| 多显示器 + 不同 DPI 缩放 | 截图坐标/尺寸错乱 | 使用 per-monitor DPI awareness，仔细处理坐标变换 |
| WebView2 未安装 | 应用无法启动 | Tauri 安装包自带 WebView2 bootstrapper |
| Canvas 标注编辑器性能 | 大量标注时卡顿 | 分层渲染，已完成标注 flatten 到底层 Canvas |
